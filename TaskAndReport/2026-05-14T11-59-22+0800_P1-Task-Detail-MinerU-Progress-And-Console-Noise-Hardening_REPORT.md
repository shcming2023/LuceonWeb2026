## Implementation Summary

### Detail Progress Parity

- `TaskDetailPage` now imports both `deriveMineruProgressLine` and `deriveTaskDisplayStatus` from `src/app/utils/taskView.ts`.
- The overview card computes:
  - `taskDisplayStatus = deriveTaskDisplayStatus(task as any)`
  - `mineruProgressLine = deriveMineruProgressLine(task as any)`
- The old overview label `消息` is now `当前进展`.
- Overview display priority is:
  1. semantic MinerU progress line;
  2. derived lifecycle display status;
  3. raw task message.
- If a semantic MinerU progress line exists and the raw task message is different, the raw message is shown as a secondary small line.
- No fabricated progress is introduced: the detail overview only displays semantic MinerU progress when `deriveMineruProgressLine` can derive it from existing task metadata.
- Terminal lifecycle truth remains authoritative because the same helper already suppresses in-flight MinerU progress lines for terminal `completed` / `review-pending` states unless the terminal no-business-signal line is explicitly valid.

### Console 503 / db-sync Noise

- Root cause localized for one clear source of repeated browser `503` resource noise:
  - `DependencyHealthBanner` polls `/__proxy/upload/ops/dependency-repair/status` every 15 seconds.
  - That status route is read-only and used to discover whether the optional supervisor is present.
  - When the supervisor is absent, `server/upload-server.mjs` returned HTTP `503` with `code=SUPERVISOR_UNAVAILABLE`.
  - The frontend caught the response and rendered status correctly, but the browser still reported repeated `503 Service Unavailable` resource errors.
- Fix:
  - The read-only status route now returns HTTP `200` with `{ ok:false, code:'SUPERVISOR_UNAVAILABLE', ... }` when the supervisor is unavailable.
  - The POST repair-action route still returns HTTP `503` for real action failures, so genuine backend/action failures are not globally swallowed or hidden.
- Initial `[db-sync]` warnings were diagnosed as separate local persistence write warnings from `src/store/appContext.tsx`. They were not changed in this task because the Task 122 evidence did not include exact write route/status details, and broad persistence behavior changes would exceed this scoped UI/progress follow-up.
