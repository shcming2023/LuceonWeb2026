## Mainline Objective

Simplify the task detail page so each processing stage exposes only its operator-facing primary artifact by default.

The page should no longer feel like it is displaying every internal file generated by MinerU or toc-rebuild. It should feel like a document-processing workbench:

```text
PDF original -> MinerU full.md -> rebuilt full Markdown
```

Technical artifacts must remain available for audit/debugging, but they should not be loaded or visually presented as main-page content.

## Product Principle

Distinguish between:

1. **Primary artifact**: the one artifact an operator needs to inspect for the stage.
2. **Technical evidence**: internal supporting artifacts for traceability, debugging, or review.

Primary artifacts:

- PDF stage: `PDF 原件`;
- MinerU stage: `full.md`;
- toc-rebuild stage: `rebuilt_markdown.md` / rebuilt full Markdown.

Technical evidence:

- MinerU `content_list.json`;
- MinerU `middle.json`;
- MinerU images folder and zip;
- toc-rebuild `readable_tree.md`;
- toc-rebuild `logic_tree.json`;
- toc-rebuild `skeleton.json`;
- toc-rebuild `provenance.json`;
- object refs, SHA, job id, metrics, event logs, diagnostics.

## Current Problem

The page already has the right staged concept, but it still exposes too many generated files as if they were equally important.

The screenshot/reporting issue is representative:

- `readable_tree.md`, `logic_tree.json`, `skeleton.json`, and `provenance.json` appear as large cards;
- they show loading states and can make the user think the core flow is broken;
- but these are not the operator's primary work object.

The operator should primarily see:

```text
PDF 原件
MinerU full.md
Rebuilt Markdown / rebuilt_markdown.md
```

Everything else should be summarized and folded under technical evidence.

## Required UI Behavior

### 1. Primary Inspection Surface

Keep the current three-lane workbench idea, but reduce it to primary artifacts only:

```text
PDF 原件 | MinerU full.md | Rebuilt Markdown
```

Requirements:

- show only `full.md` as the MinerU primary text output;
- show only `rebuilt_markdown.md` as the toc-rebuild primary full-text output when present;
- if `rebuilt_markdown.md` is missing, show a truthful empty state:

```text
暂无重建后的完整 Markdown；目录树产物已作为技术证据保留
```

- do not load or display `readable_tree`, `logic_tree`, `skeleton`, or `provenance` as main workbench panels.

### 2. Stage Output Summary

For non-primary outputs, show compact counts/status only, for example:

```text
MinerU technical artifacts: 99 files
toc-rebuild technical evidence: 7 artifacts
```

Do not list every object ref in the default main flow.

### 3. Technical Evidence Drawer

Move these items into a collapsed evidence drawer or nested disclosure:

- `readable_tree.md`;
- `logic_tree.json`;
- `skeleton.json`;
- `provenance.json`;
- SHA/object refs/job id;
- event logs;
- diagnostics;
- mock-safe operator decision preview.

The drawer may list artifact names and open links, but it must not auto-render all artifact bodies by default.

### 4. Loading Semantics

Loading or failure of technical evidence must not make the page look like the core document flow failed.

Requirements:

- primary artifact loading/failure is visible in the main inspection surface;
- technical evidence loading/failure is scoped to the evidence drawer;
- `readable_tree` / JSON evidence load failures should be secondary diagnostics, not main page blockers.
