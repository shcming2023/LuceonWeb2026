# CleanService Protocol v1

Status: Canonical protocol direction for future implementation  
Last updated: 2026-05-15  
Historical owner: Architect; role retired after 6.9.1
Mirror requirement: this file must be kept byte-identical with the approved copy in CleanService implementation repositories, starting with Mineru2Table2026.

## 1. Boundary

This protocol defines the target contract for future CleanService integrations with Luceon2026. It is not evidence that any service already implements the contract and does not authorize runtime, production, or deployment changes.

The first intended implementation is Mineru2Table2026 as `service_name="toc-rebuild"`.

## 2. Design Principles

| Principle | Requirement |
| --- | --- |
| Object references, not multipart | Primary inputs and outputs are MinIO ObjectRefs. Luceon must not make the long-term production path depend on old multipart upload routes. |
| Luceon-owned orchestration | Luceon owns material/task/version/review/audit/cost semantics. |
| Idempotent jobs | `job_id` is the idempotency key. Repeated submit of the same `job_id` must not duplicate work. |
| Explicit failure | Empty, placeholder, raw, or skeleton output must not be represented as clean success. |
| Provenance required | Completed jobs must write a `provenance.json` object with input/output hashes and service metadata. |
| Additive evolution | Protocol v1 fields are stable; future changes should add fields or define v2. |

## 3. Service Identity

Each CleanService must expose these fields from `GET /health`, job status, webhook payloads, and provenance:

| Field | Type | Example |
| --- | --- | --- |
| `service_name` | string | `toc-rebuild` |
| `service_version` | semver string | `1.0.0` |
| `protocol_version` | string | `v1` |

`service_name` must match the MinIO clean-output path segment.

## 4. Endpoints

### 4.1 `GET /health`

Returns service readiness for admission-circuit use.

Success:

```json
{
  "status": "healthy",
  "service_name": "toc-rebuild",
  "service_version": "1.0.0",
  "protocol_version": "v1",
  "checks": {
    "minio": "ok",
    "llm": "configured",
    "dependencies": "ok"
  },
  "timestamp": "2026-05-15T10:30:00Z"
}
```

If the service cannot accept work, it should return HTTP 503 with `status="unhealthy"`. Luceon may open the service admission circuit based on that result and other submit-path evidence.

### 4.2 `POST /api/v1/jobs`

Primary asynchronous job submission endpoint.

Response:

- HTTP 202 for a new accepted job;
- HTTP 200 for an idempotency hit returning existing job status;
- structured 4xx/5xx error for rejection or failure.

### 4.3 `GET /api/v1/jobs/{job_id}`

Returns current job state and output references.

### 4.4 Optional endpoints

`POST /api/v1/jobs:from-storage` may exist as an alias only if semantics are identical to `POST /api/v1/jobs`.

`POST /api/v1/extract` may remain as a synchronous/deprecated convenience path for small manual use. It must not be Luceon's long-term production integration path.

## 5. Job Request Schema

```json
{
  "job_id": "luceon-pt_xJ8kL2nP-toc-rebuild-1",
  "material_id": "sha256:a1b2c3d4e5f60718",
  "parse_task_id": "pt_xJ8kL2nP",
  "asset_version": "v1",
  "inputs": [
    {
      "role": "mineru-content",
      "source": {
        "type": "minio",
        "endpoint": "minio:9000",
        "use_ssl": false,
        "bucket": "eduassets-raw",
        "object": "mineru/sha256:a1b2c3d4e5f60718/v1/content_list_v2.json"
      }
    }
  ],
  "sink": {
    "type": "minio",
    "endpoint": "minio:9000",
    "use_ssl": false,
    "bucket": "eduassets-clean",
    "prefix": "toc-rebuild/sha256:a1b2c3d4e5f60718/v1/"
  },
  "callback_url": "http://upload-server:8788/internal/cleanservice/callback",
  "callback_secret_ref": "TOC_REBUILD_CALLBACK_SECRET",
  "options": {
    "max_workers": 1,
    "enable_virtual_anchor": true,
    "max_cost_cny": 8.0,
    "max_tokens_total": 500000
  },
  "submitted_at": "2026-05-15T10:30:00Z",
  "submitted_by": "luceon2026/upload-server"
}
```

Required constraints:

| Field | Requirement |
| --- | --- |
| `job_id` | generated by Luceon; idempotency key; maximum 128 characters |
| `material_id` | generated/owned by Luceon |
| `parse_task_id` | current Luceon task or processing instance |
| `asset_version` | required; assigned by Luceon; pattern `v\\d+` |
| `inputs[]` | at least one MinIO ObjectRef |
| `sink` | MinIO output location; prefix ends with `/` |
| `callback_url` | recommended for terminal-state delivery |
| `callback_secret_ref` | environment variable name used by the service for HMAC signing |
| `options.max_cost_cny` | service hard limit; for Mineru2Table this is `8.0` unless a later approved policy changes it |
| `options.max_tokens_total` | optional hard token ceiling |

MinIO credentials must not appear in request bodies. Services must read credentials from their environment and enforce endpoint/bucket allowlists.

## 6. Job State Schema

Allowed job states:

```text
queued -> processing -> completed
                    -> failed
queued/processing -> canceled
```

Terminal response example:

```json

  "status": "completed",

  "protocol_version": "v1",
  "material_id": "sha256:a1b2c3d4e5f60718",
  "parse_task_id": "pt_xJ8kL2nP",
  "asset_version": "v1",
```

## 7. Webhook Protocol

If `callback_url` is supplied, the service must send the full job state when entering a terminal state.

Headers:

```text
Content-Type: application/json
X-CleanService-Job-Id: <job_id>
X-CleanService-Delivery: <uuid>
X-CleanService-Attempt: <n>
X-CleanService-Signature: sha256=<hmac>
```

If `callback_secret_ref` is supplied, the service reads the referenced environment variable and signs the raw JSON body with HMAC-SHA256. Luceon must verify the signature before accepting terminal state.

Webhook delivery is at-least-once. Luceon's callback handler must be idempotent by `job_id` and terminal state.

## 8. Provenance

Each output version directory must contain `provenance.json`.

Minimum shape:

```json
{
  "schema": "luceon-provenance/v1",
  "service": {
    "name": "toc-rebuild",
    "version": "1.0.0",
    "protocol_version": "v1",
    "implementation_repo": "shcming2023/Mineru2Table2026",
    "implementation_commit": "unknown"
  },
  "asset": {
    "material_id": "sha256:a1b2c3d4e5f60718",
    "asset_version": "v1",
    "bucket": "eduassets-clean",
    "prefix": "toc-rebuild/sha256:a1b2c3d4e5f60718/v1/"
  },
  "job": {
    "job_id": "luceon-pt_xJ8kL2nP-toc-rebuild-1",
    "parse_task_id": "pt_xJ8kL2nP",
    "submitted_by": "luceon2026/upload-server",
    "submitted_at": "2026-05-15T10:30:00Z",
    "started_at": "2026-05-15T10:30:02Z",
    "finished_at": "2026-05-15T10:31:29Z"
  },
  "inputs": [
    {
      "role": "mineru-content",
      "bucket": "eduassets-raw",
      "object": "mineru/sha256:a1b2c3d4e5f60718/v1/content_list_v2.json",
      "sha256": "...",
      "size_bytes": 1234567
    }
  ],
  "options": {
    "max_cost_cny": 8.0,
    "max_tokens_total": 500000
  },
  "outputs": [
    { "role": "flooded_content", "object": "flooded_content.json", "sha256": "...", "size_bytes": 4823910 }
  ],
  "stats": {
    "tokens": { "total": 160157 },
    "cost_cny_actual": 0.184
  }
}
```

Provenance must be written after outputs are complete. If completion cannot be proven, the service must fail the job explicitly or mark the output prefix as incomplete; it must not report success without provenance.

## 9. Error Model

Errors use:

```json
{
  "error": {
    "code": "invalid_input_schema",
    "message": "Field 'material_id' is invalid",
    "details": { "field": "material_id" },
    "retriable": false
  }
}
```

Required codes:

| Code | HTTP | Retriable | Meaning |
| --- | ---: | --- | --- |
| `invalid_input_schema` | 400 | false | bad request shape |
| `invalid_input_ref` | 400 | false | input object missing or invalid |
| `forbidden_storage_target` | 403 | false | endpoint/bucket not allowlisted |
| `unauthorized` | 401 | false | API key or callback auth failure |
| `job_not_found` | 404 | false | unknown job id |
| `service_unhealthy` | 503 | true | service cannot accept work |
| `upstream_unavailable` | 502 | true | dependent service temporarily unavailable |
| `quota_exceeded` | 429 | false | cost or token hard limit reached |
| `processing_failed` | 500 | true | internal failure that may be retried |
| `processing_failed_permanent` | 500 | false | non-retriable internal failure |

When `options.max_cost_cny` or `options.max_tokens_total` is exceeded, the service must stop explicitly and return `quota_exceeded` or terminal failed job state with `retriable=false`.

## 10. Deprecated Multipart Routes

Mineru2Table's existing multipart HTTP routes may be retained for at least one version cycle, but must be marked deprecated once the protocol path exists.

Expected markers:

- `Deprecation: true`
- `Sunset: <v1.2.0-or-later-date>`
- response field `_deprecated: true`
- startup warning in the implementation repository

Luceon should not build the canonical production integration on these routes.

## 11. Observability

Each CleanService must provide:

- structured logs containing `job_id`, `material_id`, `service_name`, `event`, `level`, and timestamp;
- `/health` response within a short operational timeout;
- job events for received, started, completed, failed, and webhook delivery;
- token/cost stats for LLM-backed services;
- enough unresolved-anchor stats for operator review and audit.

## 12. Versioning

Protocol v1 is stable once accepted by Director. Future incompatible changes must use `CleanService-Protocol-v2.md` or an explicit v2 endpoint. Do not mutate v1 field meanings in place.
